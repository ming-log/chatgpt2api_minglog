# Markdown To Reviewable Image PPTX 方案

## Summary

- 新增“Markdown 生成 PPT”工具页，流程为：生成设计方案、生成可审阅图片页、确认后打包 PPTX。
- 用户可指定 PPT 页数，默认 `20` 页，允许范围 `1-100` 页。
- 用户可以逐页查看初稿图片，单独修改某页图片提示词，并只重新生成该页。
- 为保证一致性，系统锁定一份全局设计规范，每页生成时都自动追加该规范。

## Key Changes

- 后端新增 PPT 模块：
  - `POST /api/ppt/plans`：输入 Markdown 和 `slide_count`，调用当前对话接口生成指定页数的设计方案。
  - `POST /api/ppt/tasks`：按 plan 创建 PPT 图片任务，逐页生成初稿图片。
  - `GET /api/ppt/tasks?ids=...`：查询任务、每页状态、图片 URL、当前提示词和错误。
  - `POST /api/ppt/tasks/{task_id}/slides/{slide_id}/regenerate`：提交修改后的单页提示词，只重生成这一页。
  - `POST /api/ppt/tasks/{task_id}/package`：用户确认后，把当前每页图片打包成 `.pptx`。
  - `GET /api/ppt/tasks/{task_id}/download`：下载已打包的 PPTX。
- 新增 `PptTaskService`：
  - 任务按用户隔离并持久化到 `data/ppt_tasks.json`。
  - 每页保存 `title`、`original_prompt`、`current_prompt`、`final_prompt`、`image_url`、`version`、`status`、`error`。
  - 单页重生成递增版本号，保留当前最新成功图片作为最终打包来源。
- 页数规则：
  - UI 默认值为 `20`。
  - 后端强制校验 `slide_count` 必须是 `1-100` 的整数。
  - 设计方案必须精确返回 `slide_count` 页；页数不匹配时视为生成失败并提示重试。
- 提示词结构：
  - `global_style_prompt` 固定描述版式、字体风格、色彩、图形语言、留白、品牌调性和 16:9 页面规则。
  - `slide_prompt` 描述该页标题、核心信息和构图重点。
  - 实际发给图片接口的提示词为：`global_style_prompt + slide_prompt + 输出完整 16:9 PPT 页面图片`。

## Frontend Flow

- `/ppt` 页面提供 Markdown 粘贴/上传、页数输入器和生成方案按钮。
- 页数输入器默认 `20`，最小 `5`，最大 `100`。
- 方案预览展示整体设计思路、全局风格、每页标题和提示词。
- 图片审阅区展示每页生成状态、预览图、提示词编辑框、单页重新生成按钮。
- 所有页面都有成功图片后，允许点击“打包 PPTX”并下载。

## Test Plan

- 单元测试：
  - `slide_count` 默认值、边界值 `5/20/100`、非法值校验。
  - plan JSON 校验，包括全局风格和精确页数。
  - 单页 regenerate 只更新目标页，不影响其他页。
  - package 只允许在所有页面有成功图片时执行。
- API 测试：
  - 创建 plan、创建任务、查询进度、单页重生成、确认打包、下载。
  - 重复 `client_task_id` 幂等。
  - 用户隔离，不能访问其他用户的 PPT 任务。
- 前端验证：
  - 修改页数后生成对应数量的页面规划。
  - 100 页任务可正常排队、轮询、逐页审阅和打包下载。

## Assumptions

- 第一版仍生成 `.pptx`，每页是一张完整 16:9 图片。
- 图片中文字仍交给图片模型生成，不做本地文字叠加。
- 100 页会消耗大量图片额度和时间，第一版按稳定优先逐页排队生成。
